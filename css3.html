How I Built This Website

Part 5: CSS III

This time, I had two major projects: making the navigation bar collapse in mobile mode and creating a search bar. I also fixed a problem I had with my <footer> and learned a few more tidbits.

Tidbits

I learned about the line-height property which creates and sets the space between each line. I also learned how to style list items <li> in <ul> and <ol> by setting the list-style property to none;. This deletes markers such as bullet points. Similarly, if the text-decoration property is set to none;, anchor elements <a> will not be underlined as per their default setting. The text-decoration property can also be used to underline, overline and strike out (line-through;) regular text.

The overflow property specifies how to handle content that is too big to fit in the element. For example, setting the property to scroll; creates a scroll bar, while setting it to hidden; hides content that overflows its container.

The min-height property makes sure that an element does not get too small, especially on responsive websites. The element will not be displayed as smaller than the minimum height. max-height makes sure an element will not be displayed too big.

In the class, I also learned that I should be as specific with .class names as possible, and use more <div> tags in case I want to style them at a later date.

Footer

I had a problem with my footer as it wasn’t always fixed to the bottom of the page. If there wasn’t enough content to fill the entire viewport, it would just hang out in the middle of the screen.

I fixed this by giving the <body> tag a min-height: of 100vh;. This means it will always take up at least 100% of the viewport’s height. I fixed the <footer> to the bottom of the screen using position:absolute; and height:5rem. No matter how much content, it is now always fixed to the bottom of the page.

Margin in Header

As you can see in previous blog posts, I had an issue with my header: there was always some margin around the image. Even when I set margin to 0;, there was still a bit of margin between the header/image and the navigation bar. Originally, I fixed this by setting margin-bottom to a negative value, but after a bit more research I realized I had to set the image’s display to block;.

Re-Styling the Contact Form

When I originally created my contact form, I had no knowledge of CSS whatsoever. I now made sure that it looked a bit more professional.

Previously, I’ve used the fieldset element to contain my form and create a line around it. I now changed this line to border: 1px dotted darkgrey; which is the same dotted line that I am using in my sidebars. I also changed font-variant to small-caps for the titles within the contact form, and changed the font-size to 17px. Concerning the input fields, I changed their width, font-size, font-family and created a border-radius . I’ve also made sure there was enough padding and margin within and between each element. Go to contact to check out the new form and leave a message if you want.

Navigation Bar

A bigger project I had, was to create a navigation bar that would be responsive and collapse when the viewport becomes small enough. This involved my first dip into javascript.

The first step involved turning my navigation bar back into a list, and to create another list item <li> that will be the menu icon. I created the menu icon by copying Unicode from the website I previously mentioned, but there are other ways of doing it. Using a @media query, I set the icon’s display to none; when the display is bigger than 800px.

When in mobile mode, I want the navigation bar to only show the menu icon and the active category of the post. In order to do so, I assigned the list item of the category a class .active. Using the ::not selector, I set the display of all list items that are not .active to none.

I want the navigation bar to expand again when you click the menu icon. For this, I create a new class .responsive which I styled like this:

This means, categories will be displayed on top of each other, not next to each other. In order to change the original class .navbar to .navbar.responsive, I had to create a javascript function.

The function dropdownMenu()” creates a variable that contains the element with the id #dropdownClick. The function then runs an if-statement. If the element with the id #dropdownClick contains the class .navbar, the function changes the class to .navbar.responsive. Otherwise, it changes the class (back) to .navbar.

This function is activated through an onclick event. When the menu icon is clicked, the function starts to run. The icon links to a javascript void in order to trick the browser to stay on the same page.

In mobile/responsive mode, the navigation bar now only shows the active category and the menu icon, which collapses into a dropdown menu when you click on it.

Search Bar

The search bar was a project I wanted to do completely on my own, and, unfortunately, I failed. However, I still learned a lot and hopefully, once I am finished with my lesson on javascript, I should be able to fix it to some degree.

In my first step, I styled the search bar. I created two inputs: <input type=”text”> and <button type=”submit”>. I gave them the same styling: same background-color and color, same font-size, same border and border-radius and the same padding. Even though they are two different elements, they look as if they are one element. This was eventually my downfall, but I am still not sure how to fix this.

I then set the search bar’s display to none;. I created the search Icon again, display: block; but as a simple link, not as an input field. This is the icon that is originally visible when you visit the website.

Clicking on the search Icon activates an onclick event again, which calls my function expandSearchBar().

Similar to the navigation bar, the function checks for the class name of the id #searchBar and can change it to searchBar.visible which changes the search bar’s display back to block;.

Since the search icon is also a submit button, clicking on it wouldn’t automatically close the search bar. I had to create a a little cross with another onclick event that calls a new function, collapseSearchBar(). This function renames .searchBar.visible back to .searchBar.

So far, so good. However, once I worked on the search bar in responsive mode, I started having major problems. I especially struggled to keep the search icon at the top right of the navigation bar, even when I set it as float:right;, top:0; and right:0;. Once I opened the navigation bar’s dropdown menu, the search icon would be positioned at the bottom right, not top right. I thought this was because the navbar list items and the search icon were being read as part of the same element, therefore search icon was just rendered as yet another list item. I tried to fix this, by placing both the navbar and the search bar/icon in different columns: navbar in a .col-9 and search bar in a .col-3. I hoped this would portray them next to each other, not on top of each other.

This trick worked to only some extend: the search bar and icon were no longer placed at the bottom of the dropdown menu as if they were another list item, but they were now placed in the middle of the navbar (still with a float:right;). Eventually, I extended the dropdownMenu() function to include a change to .col-3 class name, which calls a different class and gives .col-3 a min-height of 324px;.

Why 324px; you might ask? Well, after a little trial and error, this turned out to be the height of .col-9 which contains the dropdown menu navbar. This made sure, that the search icon was placed at the top right of the navigation bar. Of course, this was only .col-9’s height on my device, meaning on other devices the search icon will not not be placed at the top right. So, my little hack, which took me hours to come up with, did not actually work. It has moved the search icon closer to the top, but it has not fixed my problem.

Another major problem: since the search bar consists of two different elements (<input type=”text”> and <button type=”submit”>), they kept visually separating from each other whenever the viewport became too small. I first tried to change the width of the input text field to 28vw. I thought using a relative size like this – 28% of the viewport’s width – would work on each device, but again, after testing, I found that it still broke up the search bar on other devices. My second technique involved changing the width of .col-3 to 100%; when calling the function expandSeachBar(). However, while I managed to include this change in the function, I never managed to change the width back when closing the search bar using the collapseSearchBar() function. Ultimately, I realized I had too little experience and knowledge in javascript, and decided to leave the search bar as it is right now, and I hope I will find a way to fix it when I continue my lesson on javascript.

Search Bar Input

Even if my layout had worked out, I still wouldn’t have known how to process the input. As far as I know, you need PHP-knowledge for this, but there might be other programs/languages which I hope to learn about in the future. As an alternative, I tried to create a customized search engine using google. This would use google to search through my website and display results.

I encountered two problems with this solution:

1. My website is not yet on the google index, meaning it is essentially invisible and does not show up in google searches. I have done some research on how to fix this and am planning to include this in my next blog post.

2. No matter what I tried, I always had to use a search bar created by google. I thought I could simply link to it as the <form> action attribute and they would handle the data for me (as the form handler of my contact form does), but whenever I did this, I received error messages. It seems that you have to use google’s own search bar. While you can customize its design a bit, I wasn’t satisfied by its look and ultimately deleted it.

Final Thoughts

There were many problems this time, especially when it came to the search bar. However, I am trying not to be disappointed. I have only started my journey into web development, and I think continuing my class and acquiring more knowledge will help me fix these problems (and understand them better). I am excited to learn more, but I also want to acknowledge when I am just not there yet. I spend way too much time trying to fix problems that I don’t have the skills to fix yet. Hopefully in these next weeks, I’ll gain a better understanding of when it is worth the time and effort to research and work on these problems, and when I should move on. And I find it important to chronicle these problems even if I don’t have a solution for them.

See you again next time!
